# External Agents Implementation Comparison: Zed vs Irogen

## Date: 2026-04-11

## Architecture Overview

| Aspect | Zed-main | Irogen (Current) |
|--------|----------|------------------|
| **Protocol** | `agent_client_protocol` (ACP) JSON-RPC 2.0 over stdio | Custom `message_protocol.rs` (bincode+JSON) + ACP |
| **Process Model** | Child process with stdio | `std::thread::spawn` per session |
| **Authentication** | Terminal auth + Gemini special handling | No terminal auth |
| **Session Lifecycle** | `AgentConnection` trait + `AcpThread` | `AgentManager` + `AcpStreamingSession` |
| **UI Integration** | Full agent panel, connection registry, ACP inspector | Missing |

---

## Identified Issues

### Issue 1: Missing Subagent Metadata Tracking
**Severity:** High
**Priority:** P1

Zed has complete subagent tracking mechanism:
```rust
// Zed - acp_thread/src/lib.rs
pub const TOOL_NAME_META_KEY: &str = "tool_name";
pub const SUBAGENT_SESSION_INFO_META_KEY: &str = "subagent_session_info";
```

Irogen's `AcpStreamingSession` does not implement tool name and subagent session info metadata passing, preventing tracking of subagent sessions generated by tool calls.

---

### Issue 2: Missing Terminal Auth Flow
**Severity:** High
**Priority:** P0
**Status:** ⚠️ Blocked - Needs crate upgrade + UI integration

Zed has complete terminal auth implementation:
```rust
// Zed - agent_servers/src/acp.rs
pub const GEMINI_TERMINAL_AUTH_METHOD_ID: &str = "spawn-gemini-cli";

let auth_methods = if agent_id.0.as_ref() == GEMINI_ID {
    vec![acp::AuthMethod::Agent(
        acp::AuthMethodAgent::new(GEMINI_TERMINAL_AUTH_METHOD_ID, "Login")
            .description("Login with your Google or Vertex AI account")
            .meta(meta),  // Contains terminal-auth command config
    )]
} else {
    response.auth_methods
};
```

**Required for fix:**
1. Upgrade `agent-client-protocol` to v0.10.2
2. Process `init_response.auth_methods` after initialize
3. Implement `SpawnInTerminal` UI integration for auth flow
4. Call `connection.authenticate()` when auth is required

### Issue 3: Client Capabilities Incomplete
**Severity:** Medium
**Priority:** P1
**Status:** ⚠️ Blocked - Needs crate upgrade

Zed's client capabilities include meta fields:
```rust
.connection.initialize(
    acp::InitializeRequest::new(acp::ProtocolVersion::V1)
        .client_capabilities(
            acp::ClientCapabilities::new()
                .fs(...)
                .terminal(true)
                .auth(acp::AuthCapabilities::new().terminal(true))
                .meta(acp::Meta::from_iter([
                    ("terminal_output".into(), true.into()),
                    ("terminal-auth".into(), true.into()),
                ])),
        )
        .client_info(...),
)
```

**Required for fix:**
1. Upgrade `agent-client-protocol` to v0.10.2 (current v0.9.4 doesn't have `AuthCapabilities` or `.meta()`)
2. Add `.auth(acp::AuthCapabilities::new().terminal(true))` to client capabilities
3. Add meta fields for `terminal_output` and `terminal-auth`

---

### Issue 4: Permission System Too Simple
**Severity:** Medium
**Priority:** P2
**Status:** ⚠️ Future Enhancement

Zed's permission system is more granular:
```rust
pub enum PermissionOptions {
    Flat(Vec<acp::PermissionOption>),
    Dropdown(Vec<PermissionOptionChoice>),
    DropdownWithPatterns {
        choices: Vec<PermissionOptionChoice>,
        patterns: Vec<PermissionPattern>,  // e.g., cargo build patterns
        tool_name: String,
    },
}
```

Irogen's `AcpPermissionHandler` only has basic `AlwaysAsk/AcceptEdits/Plan/AutoApprove` modes, lacking Zed's dropdown patterns and sub-patterns mechanism.

**Required for fix:**
1. Extend `PermissionOptionKind` to support `Dropdown` and `DropdownWithPatterns`
2. Add pattern matching logic for command-line arguments
3. UI components for dropdown permission selection
4. Store and persist pattern-based permission rules

---

### Issue 5: No Session List Watch Mechanism
**Severity:** Medium
**Priority:** P2

Zed's `AcpSessionList` supports watching session updates:
```rust
impl AgentSessionList for AcpSessionList {
    fn watch(&self, _cx: &mut App) -> Option<smol::channel::Receiver<SessionListUpdate>> {
        Some(self.updates_rx.clone())
    }
    
    fn send_info_update(&self, session_id: acp::SessionId, update: acp::SessionInfoUpdate) {
        self.updates_tx.try_send(SessionListUpdate::SessionInfo { session_id, update }).log_err();
    }
}
```

Irogen's `list_agent_history` only returns a one-time list without watch mechanism for session change notifications.

**Required for fix:**
1. ACP protocol must support session watching (not all agents do)
2. Long-running ACP connection for session list
3. Channel/subscription mechanism to receive updates
4. Tauri events to notify frontend
5. Frontend UI to handle real-time updates

### Issue 6: Missing Edit Tool Special Handling
**Severity:** Low
**Priority:** P3
**Status:** ⚠️ Blocked - Breaking API Changes

Zed has dedicated diff handling:
```rust
// Zed - acp_thread/src/diff.rs
// Edit tool call handling with diff representation
// ToolKind::Edit for Codex-style edit workflows
```

**Crate Upgrade Attempt Results:**
- Tried upgrading to `agent-client-protocol = "=0.10.2"` (Zed's version)
- **Failed** - Breaking API changes:
  - `kill_terminal_command` method removed from `acp::Client` trait
  - `FileSystemCapability` renamed to `FileSystemCapabilities`
  - Many other breaking changes

**Required for fix:**
1. Zed may use a private fork of `agent_client_protocol` with `ToolKind` and `Diff` types
2. Current v0.9.4/v0.9.5 doesn't expose these types
3. Need to either:
   - Find Zed's actual fork/repository
   - Or implement ToolKind detection via other means (e.g., tool name patterns)

### Issue 10: Threading Model Inefficient
**Severity:** High
**Priority:** P0 (Currently Fixing)

Irogen creates an independent `std::thread` for each session:
```rust

std::thread::Builder::new()
    .name(thread_name)
    .spawn(move || {
        let runtime = tokio::runtime::Builder::new_current_thread()...
```

Zed uses `gpui::Task` and lightweight coroutine model, more resource efficient.
